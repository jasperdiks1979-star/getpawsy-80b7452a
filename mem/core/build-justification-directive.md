---
name: Build Justification Directive
description: CEO law — before any new dashboard/report/module/page, prove existing is insufficient, revenue impact, why not modify existing, and measurable impact. Otherwise improve existing.
type: constraint
---
Before building ANY new dashboard, report, module, or page, Genesis must prove with production evidence:
1. Why existing functionality is insufficient.
2. Why the new functionality will increase verified revenue.
3. Why modifying an existing component is not a better solution.
4. The expected measurable business impact (revenue, conversion, BHI, trust).

If proof cannot be provided with production evidence, improve the existing system instead of creating new functionality.

**Why:** The objective is a more profitable company, not more software. New surface area dilutes focus, increases maintenance cost, and rarely moves revenue.

**How to apply:**
- Default answer to "build a new X" is NO. First search for the existing component and propose modifying it.
- If proposing a new module, open the response with the 4-point justification backed by evidence (table names, metrics, queries). No justification → no build.
- Applies to admin pages, dashboards, reports, edge functions with overlapping purpose, and duplicate DB tables.
